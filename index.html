What are we trying to achieve

- What is our goal? - What is the most important metric of progress? - We need contributions, so perhaps facilitating contributions is our most important activity

Setting the Scene

The Turing Way

- The Turing Way is a handbook for reproducible, ethical and collaborative data science - The book is openly licensed and developed through by the community in open collaboration. - Large number of contributors, over 470 - Diverse contributors - Educational background - Profession - Location - Language - Ethnicity - Lived experience

The Problem

Maintaining Quality

- How quality control can hinder contributions - Link to David's pages on review - Can be difficult, even off-putting for contributors

Heterogeneous Contributors

- Different tools and IDEs - To some extent you need to insist on common tools, but you don't want to create barriers - We get _very_ opinionated about text editors and IDEs

In The Turing Way

- _Very_ diverse community - Educational and professional background - Profession - Age - Location - Language - Interests - Ethnicity - Not generally software engineers - Mostly text (inspection/review is more manual, no test suite for prose) - Many have and will never build the book (and that is **fine**) - Contributors using text editors/command line (like me), through to using the GitHub web interface - **All** of these people make valuable contributions

What We Do

Why Git and GitHub Anyway?

- Why not DocuWiki, MediaWiki _etc._? - There are strong advantages to version control - Automation and CI (checks, linting, validation) - Review for **both** quality control and lending assistance - Do as we say. We promote best practice and working in the open and we mean it!

Metatext

- The book includes - contribution guide - code of conduct - processes - event organising guidance - This is all open to modification by the community, just like any other part - All of this is open and flexible, it can be what we need it to be

Recognising All Contributions

- We value and recognise all types of contributions - Not just code, not just prose - all-contributors bot used to build table - Crowdin API used to get translation contributions - Community members are encouraged to add themselves

Support

- Review is an opportunity to help - Culture of tagging/assigning experienced community members - Team members are available to help _e.g._ fixing CI, changing style - Team members will make changes themselves if it helps

Lessons

Know Your Contributors

- You need to meet them in the middle - What kind of support is most valuable to them? - The less "technical" your community is, the more hands-on support they will likely need

Building a Community Takes Effort

- Build it and they may not come

Trust in CI

- No arguments about "it works for me" - There is a healthy level of quality control and you may need to iterate to find it

Version Control is not Optional

- The advantages are too strong - It is worth the cost of on-boarding and providing support

Be Flexible

- Remember what the objective is - It is better to bend the rules a little or do something that "isn't my job" if it gets a good contribution merged - A spelling or grammatical mistake could even make a great first issue

But Know When to be Strict

- Don't merge changes which break things - Don't knowingly introduce problems which you expect someone else to fix - Keeping quality high introduces some pain, but saves a lot of pain in the long run
